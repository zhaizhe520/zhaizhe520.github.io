Vue 3 则是最长递增子序列 + 统一遍历

但是，如果用户在前端玩了一把“极端的乱序大洗牌”呢？

比如一个商品列表，用户点了一下“价格从低到高排序”，整个列表的顺序瞬间全乱了。

这时候，Vue 2 的 4 个指针就会全部抓瞎，两头对不上，只能苦哈哈地退化成去哈希表里一个个查、一个个在页面上无脑移动 DOM 节点。

DOM 操作是整个浏览器里最贵的硬件开销。频繁地去移动一个 DOM 节点，会触发浏览器的重排（Reflow），非常卡。

能不能找到一种算法，算出在乱序大洗牌时，有哪些节点其实可以“原地躺平不动”，只去移动那些非动不可的节点？

这就引出了最长递增子序列。

找出原地躺平不动的钉子户

[0, 1, 2, 3] —>[2, 0, 1, 3]—–>[0 , 1, 3]为最长没动的递增子序列

Vue 3 完整的乱序 Diff 过程，其实是一套极其丝滑的组合拳：

  1. 统一遍历（预处理）

大厨先从左往右遍历、再从右往左遍历。把头部一模一样的节点、尾部一模一样的节点直接复用。这步叫“前后预处理”，能直接干掉 80% 没变动的场景。

  1. 建立新旧映射表（捞出乱序核心）diff算法

处理完头尾后，中间如果剩下一堆乱序的节点，Vue 3 会用一个数组记录下这些新节点在旧列表里的位置。

  1. 求出最长递增子序列（算出不动的主干）

对这个映射表数组，运用动态规划或二分查找的数学公式，算出最长递增子序列。

这个子序列，就是整个页面上的“稳定大后方（不动产）”。

  1. 倒序移动（查缺补漏）
    大厨从后往前遍历乱序区，如果发现当前节点的下标在“最长递增子序列”里，大厨直接打个手势：“你别动，跳过！”

如果发现不在子序列里，大厨才动手：“起来，把你移动到指定位置。”

Vue 2 在运行时（浏览器里）是有点“笨”的。当数据变了，它会把新旧虚拟 DOM 全通篇 Diff 比对一遍，哪怕你的 HTML 里 90% 都是绝对不会变的死文字。

Vue 3 引入了极其强悍的编译期静态分析（Static Analysis），在把 .vue 文件打包编译成 JS 的时候，就已经把浏览器要干的活给提前干完了。

响应式的降维打击：从“死脑筋拦截”到“全局影分身”

Vue 2 的死穴：Object.defineProperty（老实人挨个盯着）

Vue 2 的响应式，是大厨拿着放大镜，去把你的 data 对象里的每一个属性用 for...in 遍历一遍，给每个属性强行加上getter 和 setter。

致命缺陷 1（性能差）：如果后端传给你一个 1 万条数据的超级大对象，Vue 2 一上来就会不管三七二十一，深度遍历这个大对象的所有格子，把内存直接吃满，导致首屏加载巨卡。

致命缺陷 2（测漏）：大厨初始化时只盯着现有的属性。如果你在代码里突然写一句 this.obj.newKey = 'Murasame'（新增属性），或者用下标修改数组，Vue 2 的大厨根本“看漏了”，页面完全不更新，你必须卑微地去用 this.$set()。

Vue 3 的外挂：Proxy（全局雷达防空网）

Vue 3 彻底抛弃了挨个拦截的笨方法，直接利用了原生 ES6 的 Proxy（代理）。

大厨不再去碰原始数据，而是在原始对象的外面，套了一层透明的防空网（代理对象）。

真香定律：管你对象里有一万个属性还是新增属性，任何人想去读取或者修改这个对象，都必须经过外面这层防空网的拦截（get/set 钩子）。

降维打击：Vue 3 实现了惰性监听（Lazy Initialization）。只有当你在页面里真正用到深层数据时，它才会去给那那一层加响应式。

首屏速度直接飞起，而且天生完美支持动态新增属性和数组下标修改，this.$set 直接被扫进了历史垃圾堆！

编译期的“降智打击”：把活在编译阶段干完！

Vue 2 在运行时（浏览器里）是有点“笨”的。当数据变了，它会把新旧虚拟 DOM 全通篇 Diff 比对一遍，哪怕你的 HTML 里 90% 都是绝对不会变的死文字。

Vue 3 引入了极其强悍的编译期静态分析（Static Analysis），在把 .vue 文件打包编译成 JS 的时候，就已经把浏览器要干的活给提前干完了。

补丁标记（Patch Flags）：睁眼瞎变成了精准打击

Vue 2：组件更新时，虚拟 DOM 会老老实实对比 <div>、再去对比 <h1>（发现没变）、再去对比 <p>。

Vue 3：编译时，大厨一眼看穿<h1> 是死文字，<p> 是动态的。于是它在生成的虚拟 DOM 节点后面，偷偷打了一个数字补丁标记（Patch Flag），比如标记为 1（代表只有文本会变）。

当页面更新时，Diff 算法看到 <h1> 直接跳过连看都不看，直接闪现到<p> 进行精准更新。

静态提升（Static Hoisting）：不给内存添乱

在 Vue 2 里，只要组件一更新，所有的虚拟 DOM 节点都会在内存里重新被 createVNode 创建一遍，哪怕是那个永远不变的 <h1>。

Vue 3：大厨直接过河拆桥。把所有绝对不会变的静态节点，打包提升到组件外面去变成全局常量。

无论你组件更新 1 万次，这个 <h1> 节点在内存里永远只被创建 1 次，直接把浏览器的内存垃圾回收（GC）压力降到了最低。

reach ability ansinsty

就是在内存创建一次才能用虚拟dom和diff算法来找不同实现dom操作，不同的才会动，相同的不动，最长自增子序列

代码组织：从“按菜单抓药”到“按模块拼乐高”

也就是你天天看得到的 Options API（Vue 2） 变成 Composition API（Vue 3）。

Vue 2（Options）：逼着你把代码拆散在 data、methods、computed 里面。如果你要写一个“黑猫”的功能和一个“白狗”的功能，你得把黑猫的变量写在 data 头部，黑猫的方法写在 methods 尾部，白狗的交织在中间。代码长了之后，上下反复横跳，极其痛苦。

Vue 3（Composition）：允许你把“黑猫”的所有变量、方法揉成一个独立的 useBlackCat() 函数，把“白狗”揉成 useWhiteDog()。最后在 setup 里面像拼乐高一样组装。这不仅是写法变了，而是让前端真正具备了和后端一样的、完美的“高内聚低耦合”的代码抽象能力。

Vue 2 到 Vue 3 的进化：总结

2 代更像是一个精巧的、老老实实工作的手工工坊，用着双端指针、defineProperty 挨个打补丁。

3 代则直接升级成了工业化自动流水线。它用 Proxy 编织防空网，用 最长递增子序列 钉住主干，用 编译期优化 提前给 DOM 打上指路牌，用 Composition API 解放了生产力。
