数据库里单独存一个“缩略图”或者“图片高宽比”来优化

不晓得这是怎么做到的

在数据库里单独存“缩略图 URL”或者“图片宽高比（Aspect Ratio）”，是商业项目里用来解决“网页排版塌陷”和“图片加载慢”的终极杀招。

结合你现在用的 WordPress + Vue 3 架构，我们来拆解一下它是怎么一步步实现的。

一、 为什么要存“图片宽高比”？（解决闪烁与塌陷）
你之前遇到“骨架屏消失后，图片还没加载出来”的情况，如果你的卡片容器没有固定高度，页面就会塌陷成一个扁平的方块。等图片啪地一下加载出来，卡片又突然被撑大。这种视觉上的猛烈跳动，叫做 CLS（累积布局偏移），体验非常差。

解决思路：
如果数据库提前告诉前端：“这幅图的宽高比是 16:9（也就是 1.777）”，Vue 3 就可以在图片还没下载完时，提前用 CSS 把这个卡片的格子撑开到完美的比例。

1.数据库表结构设计
在数据库里，我们通常会在文章表（或者专门的媒体表）里加两个字段：

SQL
– 假设我们在 wp_posts 或者自定义表里加字段
ALTER TABLE wp_posts ADD COLUMN thumb_url VARCHAR(255); – 缩略图小图链接
ALTER TABLE wp_posts ADD COLUMN aspect_ratio DECIMAL(4,2); – 宽高比，比如 1.77
2.前端 Vue 3 怎么利用这个值？

效果： 骨架屏的大小和后面出来的图片严丝合缝，一像素都不带抖动的。骨架屏消失时，图片已经在这个框里默默渲染好了，视觉效果极度舒适。

为什么要存“缩略图”？（解决加载慢）

WordPress 默认上传一张高清大图时，服务器后台会自动把它裁剪成好几种规格（比如缩略图 Thumbnail、中等 Medium、大图 Large）。

如果你在前端列表页（那 8 个盒子）里直接加载几兆（MB）的原图，网页肯定卡死。正确的做法是：列表页存小图（几十 KB），详情页点进去再看大图。

这是怎么实现的？
后端/CMS 负责生成：
当你通过 WordPress 后台上传一张名为 murasame.png（2MB） 的图片时，WordPress 会自动在服务器生成一张 murasame-150x150.png（20KB） 的小图。

存入数据库：
WordPress 会把这张小图的 URL 写入数据库的 wp_postmeta 表中。

前端 Axios 获取：
你在 Vue 3 里调接口时，不要去拿 guid（那个通常是原图），而是去拿 WordPress 专门开放出来的缩略图字段（比如 featured_media_src_url）。

🛠️ 怎么在你的项目里落地？
因为你现在用的是 WordPress 作为 Headless CMS（无头后端），你其实不需要自己去 MariaDB 里手动建表和算宽高比，WordPress 已经帮你算好了。

你昨天用 Axios 抓取 /wp-json/wp/v2/posts 时，可以在请求的 URL 后面加一个参数：?_embed。

JavaScript
// 示例：昨天的 Axios 请求加上 _embed
axios.get(‘/wp-json/wp/v2/posts?_embed’)
加上这个参数后，返回的 JSON 数据里会多出一个 _embedded 字段，里面贴心地包含了：

各种尺寸的缩略图 URL（thumbnail / medium）

图片的原始宽度（width）和高度（height）

拿到宽高之后，你直接在前端算一下：const ratio = width / height，然后把它赋给你的 Vue 盒子。

你可以看一下你昨天拿到的文章 JSON 数据里，有没有包含图片高宽的信息？如果有的话，今天直接用 CSS 的 aspect-ratio 连上去，保准你的骨架屏体验能再提升一个档次！
