内存泄漏

内存泄漏的发生：你以为这个组件已经死了（比如 Vue 路由切走了，组件销毁了）

但由于某种原因，全局的 window 还牵着它的一根线，系统误以为它还活着，就不敢杀它，它就漏了。

详细说场景前，你得先知道 Vue（浏览器）和 Android（JVM）目前通用的垃圾回收算法：可达性分析算法（Reachability Analysis）。

系统会从一个根节点（叫做 GC Roots，比如全局变量 window、当前执行栈等）开始往下摸。

只要顺着这条线能找到的对象，就是“活着”的。

顺着线找不到的对象，就是“死了”的（垃圾，会被清理）。

遗忘的定时器（setInterval / setTimeout）

这是最常见的翻车现场。

场景：你在组件 A 里开了一个定时器，每秒刷新一次数据。当路由切换到组件 B 时，组件 A 看起来在屏幕上消失了（销毁了）。

原因：如果你没有手动清除定时器，浏览器的定时器模块还一直在运行，并且它内部的闭包依然死死抓着组件 A 的变量。

解法：在 Vue 的 beforeUnmount 或 onBeforeUnmount 生命周期里手动清除。

被遗忘的全局事件监听器（Event Listener）

场景：为了做某些特效，你在组件里监听了全局窗口的滚动：window.addEventListener(‘scroll’, handleScroll)。

原因：当组件销毁时，这个监听器是挂在 window（全局根节点） 上的。window 不死，handleScroll 就不会死，而 handleScroll 又是组件内的方法，导致整个组件都无法被回收。

解法：离开页面时必须 removeEventListener。

闭包（Closure）的滥用

场景：一个外层函数返回了一个内层函数，内层函数死死咬住外层函数的局部变量。

脱离 DOM 的元素引用（Detached DOM）

场景：你在 JavaScript 里用一个变量存了一个按钮：const myBtn = document.getElementById(‘btn’)。后面你通过代码把这个按钮从 HTML 页面上删除了。

原因：虽然页面上没有这个按钮了，但由于你的 JS 变量 myBtn 还指着它，这个按钮就会变成“孤魂野鬼”（Detached DOM），一直留在内存里。
