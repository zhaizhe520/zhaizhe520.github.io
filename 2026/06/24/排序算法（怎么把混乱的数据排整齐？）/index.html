排序算法（怎么把混乱的数据排整齐？）

冒泡排序（Bubble Sort）—— 笨重但老实的“排排坐”

最长递增子序列也算是排序算法

🎬 物理画面：鱼吐泡泡

第一轮：大厨从最左边第一个人开始，让相邻的两个人两两比身高。如果左边比右边高，两人就交换位置。这样一路比过去，你会发现，全场最高的那个人，就像水底的一个大泡泡一样，一轮比完之后，被硬生生地“顶”到了最右边。

第二轮：大厨重新回到最左边，重复刚才的动作。这一轮比完，第二高的人又被“顶”到了倒数第二的位置。一直重复 n 轮，队伍就排好了。

📊 性能体检表:

时间复杂度：O(n^2)。因为大厨要玩双重 for 循环。外面要跑 n轮，里面每轮还要挨个比对。

如果数据有 1 万条，大厨得比对差不多 1 亿次！在工业级生产环境里，这种算法就是个垃圾，会被直接开除

空间复杂度:O(1)。因为大厨完全是在原始数组里“原地挪位置”（交换变量），不需要额外开辟新的 Map 或者数组空间，非常省内存。

快速排序（Quick Sort）

🎬 物理画面：找个“基准”两边倒

1、找个参照物（Pivot）：大厨随便在数组里挑一个数字（比如最中间的那个数字，叫它“基准”）。

2、阵营大划分（统一遍历）：大厨让其他所有数字过来跟这个基准比大小。小的全部扔到左边，大的全部扔到右边。

3、这时候，虽然左边和右边各自内部还是乱的，但大局观已经定了——左边的所有人，绝对都比右边的小！而那个基准数字，已经精准地坐在了它这辈子最正确的最终位置上！

4、递归套娃（文件夹里套文件夹）：

5、循环以上排序直到排序正确

📊 性能体检表

时间复杂度：平均 O(n \log n)这是一个极其高效的跨越。为什么是 O（log n）？因为大厨每次都在玩“对折划分”。1000 万条数据，冒泡排序要卡死，快排大厨几毫秒就给你切瓜砍菜般搞定了！

快排最坏情况:选了个最大的，直接变成n平方

TimSort 算法

TimSort 的核心思想是：人类在真实业务里写的数据，绝对不可能是纯随机的乱序，往往里面已经有一部分片段是“局部有序”的。

有序小片段进行归并

自动切块：寻找“顺风车” 片段（Run）

大厨拿到数组后，先从左往右扫描。

如果发现有一段数字已经是排好序的（比如 [3, 5, 8, 9]），大厨就会把这一段打包成一个“顺风车片段”（专业术语叫 Run）。

如果发现有一段是倒序的，大厨顺手顺脚地帮它当场反转过来，也变成一个正序的 Run。

小数据嫌弃：直接降维打击（插入排序）

如果大厨发现这个 Run 片段太短了（比如少于 32 个元素），大厨连快排、归并这种大招都懒得放。

它会直接掏出一个叫

插入排序（Insertion Sort）的精巧算法
，把后面零散的数字像插扑克牌一样，极其快速地插进这个 Run 里面，把它补满到一个标准长度。

大局观合体：智能骨牌合并（归并排序） 归并（Merge）逻辑。

当整个数组被大厨切成了一个个有序的 Run 片段之后，大厨掏出真正的王牌 ——

归并（Merge）逻辑

大厨把这些 Run 压入一个执行栈里。每当发现栈里有相邻的两个 Run，大厨就会把它们合二为一。

🛠️ 合并的物理画面：

两个已经各自排好序的队伍，合并成一个大队伍，速度是极其惊人的（只需要O(n)级的光速遍历）。

所有有序小片段全部融合成一个巨大的、完美有序的最终数组

TimSort 的逆天战绩（体检表）

V8 引擎用 TimSort 取代了老算法之后，带来了两个跨时代的震撼效果：

  1. 战绩：完美的稳定性（Stable Sort）

Vue 2 时代的前端痛点：由于老算法不稳定，如果你对一组商品先按“销量”排序，再按“价格”排序，第二次排序会把第一次销量的相对顺序给彻底打乱。彻底打乱之后diff算法和双端队列重新排布dom

Vue 3 / 现代前端时代：TimSort 天生是绝对稳定的。相同权重的节点，排序后绝对不会互换位置。这给虚拟 DOM 的稳定复用提供了天然的底层保障！ run片段小的有序片段不动 静态编译 操纵虚拟dom 实现时间复杂度大幅度消减

  1. 速度：在真实世界里接近神级别

最坏情况（纯乱序）：它能死死守住 $\mathbf{O(n \log n)}$ 的底线，绝对不会像快排那样翻车退化成 $O(n^2)$。

最好情况（数组本来就差不多排好了）：它的时间复杂度能达到恐怖的 $\mathbf{O(n)}$！大厨只需要顺着扫一遍，直接收工。
