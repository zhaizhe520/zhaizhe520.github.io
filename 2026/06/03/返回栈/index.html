快速连击（Double Click / Fast Tap）导致的路由疯狂压栈

🚨 还原你的 Bug 现场：为什么按钮“消失”了？

因为你点的速度太快了。在首页那个 Button 还没来得及切走的一瞬间，你啪啪连点了两下。

第一下点击：执行了 navController.navigate(“detail”)，路由开始往详情页跳。

第二下点击：因为速度极快，首页按钮还没完全从屏幕上消失，你又触发了一次 navController.navigate(“detail”)！

结果就是：你的路由栈里被塞进去了两个一模一样的详情页（Detail -> Detail -> Home）。

当你退出来的时候，你以为你点返回应该回到首页，但实际上它只是从“第二个详情页”退回到了“第一个详情页”。而因为你的详情页长得一模一样，你会误以为界面卡住没动。如果你在这个时候疯狂乱点，把路由栈彻底点空或者点乱了，界面上的组件（比如按钮）就会因为状态错乱而直接“消失”或者白屏！

在前端写 Vue Router，如果你疯狂连击一个 router.push(‘/detail’)，控制台其实会直接抛出一个红色警告：NavigationDuplicated: Avoided redundant navigation（避免了重复导航）。但安卓的 Compose Navigation 默认比较耿直，你叫它跳几次，它就傻傻地去压几次栈，结果把自己搞崩了。

防抖逻辑贴进去:

🛠️ 企业级大厂是怎么解决“快速连击”的？

在商业项目里，绝对不能允许用户连击把路由点穿。通常有两种前端同学最熟悉的解法：

解法一：防抖 / 节流（Debounce）—— 最彻底

限制按钮的点击频率。点击一次后，接下来的 500 毫秒内不管用户怎么疯狂点，直接忽略掉。

系统时间戳对比。只要两次点击的间隔小于 500 毫秒，直接闭嘴，不执行任何回调！

我们利用 Kotlin 的扩展函数，给 Android 所有的 UI 组件（Modifier）直接外挂一个防抖点击事件。

解法二：标准大厂路由单例配置（Launch Single Top）

告诉路由总管：“如果我想去的这个页面（比如 detail）已经处于当前路由栈的顶部了，就不要再重复创建它了！”

打开你的 NavGraph.kt，把首页跳转的那一段加上一个 launchSingleTop = true 的配置：

launchSingleTop = true

返回栈逻辑

防狂点特效药 —— launchSingleTop（最推荐）

navController.navigate(“my_screen”) {
// 🌟 终极特效药：开启单顶模式
launchSingleTop = true
}
会布局重复绘制

popBackStack（回滚/弹栈）

navController.popBackStack(“home”, inclusive = false)

它的意思是：从最顶上的页面开始往下清理，一直清理到 “home” 为止，但是【不包含】”home” 自己（即保留 home）。
