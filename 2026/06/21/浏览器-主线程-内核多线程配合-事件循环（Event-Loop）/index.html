浏览器(主线程+内核多线程配合 + 事件循环（Event Loop）)

浏览器五大线程类比餐厅角色

员工职位（线程名称） 岗位职责（核心功能） 对应餐厅里的角色
JS 引擎线程 负责解析和运行 JavaScript 代码（比如 V8 引擎）。它就是那个单线程！ 大厨（全餐厅只有这一个大厨，切菜、炒菜全靠他，一次只能干一件事）
GUI 渲染线程 负责渲染浏览器界面，解析 HTML, CSS，构建 DOM 树和布局。 装修队 / 传菜员（负责把盘子摆得漂亮，把餐厅装修好）
定时器触发线程 负责管 setTimeout 和 setInterval 的倒计时。 沙漏计时员（专门帮大厨盯着蒸鱼的时间）
异步 HTTP 请求线程 负责处理 Ajax、fetch 等网络请求。 外购采购员（专门去海鲜市场帮大厨买食材）
事件触发线程 负责管用户的点击（click）、鼠标滚动等事件。 前台迎宾（盯着哪个顾客拍桌子叫服务员）

执行栈

JS 引擎用来追踪“当前程序正在执行哪个函数”的一种数据结构。

初始状态：执行栈是空的。

全局上下文入栈：JS 引擎开始读取整篇代码（这叫全局执行上下文 global()），压入栈底。

调用 funcA()：遇到 funcA()，把它压入栈顶。此时栈内：[global, funcA]。

调用 funcB()：在 funcA 内部遇到了 funcB()，把 funcB 压入栈顶。此时栈内：[global, funcA, funcB]。

执行 funcB：此时 funcB 在最顶端，打印 ‘B工作完了’。funcB 执行结束，弹出执行栈。此时栈内：[global, funcA]。

继续执行 funcA：回到 funcA 刚才停下的地方，打印 ‘A工作完了’。funcA 执行结束，弹出执行栈。此时栈内：[global]。

结束：全局代码跑完，global 弹出，执行栈再次变空。

异步处理

遇到同步任务：大厨自己当场做完。

遇到定时器（setTimeout）：大厨把沙漏往“定时器触发线程（计时员）”怀里一塞：“你帮我倒计时 3 秒，到了叫我！”然后大厨继续炒下一个菜。

遇到网络请求（Axios / Fetch）：大厨把订单往“异步 HTTP 请求线程（采购员）”手里一塞：“你去服务器买个数据，买到了叫我！”大厨转头又去炒别的菜。

微任务(MicroTask)

由 JS 引擎自身发起的异步任务。它们的优先级极高，只要同步代码一执行完，立刻插队全部执行掉，执行完微任务才会去执行下一个宏任务。

Promise.then() / .catch() / .finally()

Vue 3 的 nextTick()

MutationObserver

宏任务(MacroTask)

由浏览器或宿主环境发起的异步任务。它们通常比较重，需要排队等候。

setTimeout / setInterval

postMessage

DOM 事件（如用户点击 click、滚动 scroll 后的回调函数）

Ajax/Fetch 网络请求完成后的回调

事件循环（Event Loop）

微任务队列（Micro-task）：存放 Promise.then 的回调、MutationObserver（像 VIP 顾客插队）。

宏任务队列（Macro-task）：存放定时器到期、点击事件触发的回调函数（像普通顾客排队）。

第一步（执行栈）：大厨（JS主线程）先把手头所有的同步代码（主线程执行栈）全部炒完。

第二步（微任务）：主线程空闲了。大厨走到厨房门口，先看“微任务队列（VIP通道）”。如果里面有 Promise.then 在排队，大厨会一口气把所有的微任务全部做完，直到 VIP 通道清空。

第三步（渲染）：大厨看一眼装修队（GUI渲染线程）：“页面需要更新吗？”如果要更新，装修队此时入场开始刷新网页（Render）。

第四步（宏任务）：大厨再走到厨房门口，看“宏任务队列（普通通道）”。注意！大厨非常抠门，他只拿出一个宏任务出来做（比如执行一个 setTimeout 的回调）。

第五步（无限循环）：做完这一个宏任务后，大厨立刻回到第二步，重新去检查 VIP 通道（微任务）。

注意

JS 引擎线程和 GUI 渲染线程是互斥的！ 也就是说，大厨在炒菜（JS 执行）时，装修队（GUI 渲染）必须停工；装修队在刷墙时，大厨必须熄火。这就是为什么如果你的 JS 写了死循环，网页当场就会“白屏”或卡死，因为 GUI 线程拿不到控制权，没法刷新页面！

知道这些开始在前端处理海量数据吧

耗时的大数据计算：丢给 Web Worker

如果前端遇到了不可避免的超大数据处理（比如前端解析几十兆的 Excel、处理高清图像滤镜）：

做法：千万别在主线程硬扛！利用浏览器内核的多线程特性，开辟一个 Web Worker 线程，把脏活累活丢给它。

效果：Web Worker 在后台默默计算，算完了把结果传回来。大厨（主线程）在前面依然可以轻轻松松地配合 GUI 渲染，页面丝滑不卡顿。

分片渲染（Time Slicing）

如果要往页面上一次性渲染 10 万条 DOM 节点（比如超大的树状图）：

如果一次性暴力渲染：JS 会执行很久，GUI 也会因为渲染量太大卡死。

聪明的解法（React Fiber 的核心思想）：

利用 requestAnimationFrame 或者 requestIdleCallback,实现时间分片

把 10 万条数据拆成 100 组，每组 1000 条。大厨炒完一组（渲染 1000 条），主动停火，把控制权让给装修队（GUI）喘口气、刷个墙，下一帧再接着炒下一组。

Web Worker（多线程外包） | Time Slicing（主线程分期）

维度 Web Worker（多线程外包） Time Slicing（主线程分期）
解决的痛点 耗时极长的 纯数据计算、加密、大文件解析。 耗时极长的 DOM 渲染、大量视觉元素的频繁计算。
工作地点 完全独立的 新线程，和主线程并行运转。 依然在 主线程，只是把长任务切成无数小碎块。
能否操作 DOM 绝对不能。 可以，它本身就是为了配合 DOM 渲染而生的。
本质思想 充分利用多核 CPU 的硬件性能（空间换时间）。 极致的时间管理大师，欺骗人类的眼睛（时间换流畅）。
