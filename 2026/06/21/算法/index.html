算法 数据结构 这其实是人类逻辑思维的最高浪漫

时间复杂度

随着你要处理的数据量变大，你的算法执行所需要的时间是怎么增长的。

为什么不用“秒”来算时间？

你可能会问：“为什么不直接用手表数一数，这个算法运行了多少秒呢？”
因为换一台配置更高的电脑，或者换一种编程语言，运行秒数就会改变。为了公平且客观地评价一个算法，科学家们决定：不数“秒数”，而是去数“代码执行的次数（步数）”。

这就引出了大名鼎鼎的 大O表示法（Big O Notation）。

时间复杂度：$O(1)$ —— 叫 常数阶。不管数据规模多大，速度都一样快。

时间复杂度：$O(n)$ —— 叫 线性阶。钥匙数量（数据量）翻倍，你花的时间也翻倍。

时间复杂度：$O(n^2)$ —— 叫 平方阶。如果只有2把钥匙，试 4 次；如果有10把钥匙，就要试 100 次！

数据量变大一点，时间就会暴涨。

空间复杂度（Space Complexity）处理数据的工作台

永恒真相：空间换时间

要在内存条（RAM）里额外多占多少个格子。

$\mathbf{O(1)}$:绝不占地方的老实人（常量空间）

额外占用的内存空间是恒定不变的，跟数据量有多大毫无关系。

$\mathbf{O(n)}$:海量复制的暴发户（线性空间）

const map = new Map();一个新的寄存

线性查找 / 二分查找：非常抠门，不想多占一丁点内存（空间 $O(1)$），代价就是大厨必须多跑几步、多花点时间（二分时间 $O(\log n)$，线性时间 $O(n)$）。

哈希查找（Map）：超级败家，一上来就要在内存条里圈一大块地盘（空间 $O(n)$），但回报是极其恐怖的——大厨找任何东西都只需要 1 步（时间 $O(1)$ 闪现）。

栈（stack） 深度优先 后进先出 逻辑行为结构

栈（Stack）就是“剥洋葱，或者堆盘子，后来的反而先走（后进先出 - LIFO, Last In First Out）

进栈（Push）：球只能一个一个往里塞。最先塞进去的球，被死死压在最底下。

出栈（Pop）：想拿球？对不起，你只能从最上面的开口拿。最后塞进去的那个球，最先被拿出来。

栈在 JS/DOM 进程里的第一大体现：执行栈（Call Stack）

1.调用 init() —> init 压入栈底 [ init ]

2.进入 init，发现要调用 renderDOM() ——> renderDOM 压入栈顶 [ init, renderDOM ]

3.进入 renderDOM，发现要调用 createButton() —-> createButton 再次压入栈顶 [ init, renderDOM, createButton ]

4.此时栈顶是 createButton，大厨执行打印，执行完毕。

5.createButton 弹出栈——> [ init, renderDOM ]

6.renderDOM 发现没事干了，弹出栈—–> [ init ]

7.init 执行完毕，弹出栈 [ ]（栈空，主线程闲下来了，Event Loop 准备去看微任务）。

栈在 DOM 进程里的第二大体现：DOM 树的深度优先遍历（DFS）

深度优先搜索（DFS），而 DFS 的底层灵魂就是“栈”。

场景还原：
假设有这样一个 DOM 结构：

把根节点 <div> 放进栈里。

弹出栈顶的 <div>，开始处理它。接着把它的孩子们（<p> 和 <button>）放进栈里。

此时栈顶是 <p>，弹出 <p>，优先处理 <p> 里面的内容（这就是“深度优先”，先往深了走，而不是先管旁边的 <button>）。

当 <p> 这一条分支全部处理完了（出栈空了），大厨一回头，发现栈里还躺着 <button>，再把它弹出来处理。

ctrl + Z\Y 链表+栈

正常操作：一路往前

你依次访问了网页 A -> B -> C。

大厨把它们一个个压入 历史栈。

触发回溯：点击“后退”或按 Ctrl + Z

你想回溯？太简单了！大厨把历史栈顶的 C 弹出（Pop），顺手把它压入（Push）到右边的“前进栈”里。

（当前在历史栈顶：B 网页。右边记住了你刚刚退出来的 C）

看！现在你退回到了 B。如果你再点“前进”（Ctrl + Y），大厨就把右边的 C 弹出来，重新塞回左边。完全不需要修改任何指针！

核心杀招：在历史中“改写未来”

现在你站在 B 网页上，突然点了一个全新的链接 D。

这时候，右边那个桶里的 C（原本的未来）就变成了“平行宇宙里被抛弃的历史”。

大厨怎么处理？非常无情：

直接把右边的“前进栈”一把清空（C 彻底消失，不需要断开复杂的链表线）。

把新网页 D 压入左边的“历史栈”。

（当前在最新网页：D。过去的 C 空间直接被清空释放）

队列 (queue) 逻辑行为结构

两头通透

队列的规则极其简单纯粹：先进先出（FIFO - First In First Out）。

入队（Enqueue）：新来的人，只能老老实实从队尾（Rear）排进去。

出队（Dequeue）：办完事的人，只能从队头（Front）离开。中间的人绝对不能插队，后面的人也绝对不能翻墙。

消息队列

还记得我们上次聊到的 12306 抢票、或者电商“双十一”秒杀吗？

几百万人同时在 0 点点击“提交订单”。如果后端服务器直接去扛这几百万个请求，数据库当场就会被冲垮（爆库）。

解决办法：后端在最前面架设一个“大坝”，这个大坝叫消息队列（比如 RabbitMQ, Kafka）。

几百万个请求像洪水一样涌过来，服务器不急着处理它们，而是先把这些请求一股脑全部塞进消息队列里排队。

后端的业务服务器像个无情的流水线工人，从队列的头部，每秒钟只拉取 1000 个请求出来慢慢处理。

效果：后面的用户可能会看到页面显示“订单处理中，请稍候…”，但由于队列在中间起到了“削峰填谷”的缓冲作用，服务器稳如泰山，绝不会崩溃。

双端队列（Deque）全名: Double-ended Queue

Vue 3 则是最长递增子序列 + 统一遍历
这是什么玩意

Vue 2 核心的 Diff 算法中，为了用最快的速度对比新旧两个虚拟 DOM 列表的变动，它会使用双端队列的思路：同时从新旧列表的“头部”和“尾部”向中间双向遍历对比。

这种“两头包抄”的算法，比傻乎乎地从头到尾一个个对，速度要快得多

内存里的计算，比浏览器的排版渲染快太多太多了

眼里盯着新旧两个队列的 头指针 和 尾指针

你如果在控制台打印一个空 <div>实例，会发现它身上密密麻麻挂着几百个属性。

更致命的是，直接修改真实 DOM 会触发浏览器的重排（Reflow）和重绘（Repaint），

虚拟 DOM（Virtual DOM）

当数据发生变化时（比如列表中间删除了一个名字）：

框架会在内存里生成一棵全新的虚拟 DOM 树。

拿新虚拟 DOM 树和旧虚拟 DOM 树进行肉眼可见的“找茬游戏”。

这个“找茬”的对比算法，就叫做 Diff 算法！

找出到底哪一小块发生变化后，大厨（主线程）再去精准地只修改那一只真实 DOM。

哈希表（Hash Table，也叫散列表） 逻辑结构？内存压根没有，人类纯对应造出来的

哈希表，其实是用“数组”作为底座，通过“哈希函数”算出来的下标，遇到哈希冲突会加链表进行串联拼所出来的一种极其聪明的逻辑结构！

肉身是物理的，灵魂是逻辑的

哈希表在内存里，本质上就是一排连续的数组。

哈希函数（Hash Function）。查找纯储结构里面的数据的函数

ES6 里的 Map 和 Set

现代前端如果要做高性能的去重、或者存键值对，都会用 new Map() 或 new Set()。

Set 里的值不能重复，它怎么知道重复了？

它就是在往哈希表里塞的时候，发现对应的位置已经有人了，直接拒绝，从而实现了 O(1) 级别的极速去重。

map 寄存 纯储数据时候发现hash值和下标都一样的时候直接

为什么 Vue 的 v-for 循环里必须写 :key？

当 Vue2 用 Diff 算法对新旧列表进行两头包抄“找茬”时，如果遇到了乱序:

Vue2 会把旧节点的 key 收集起来，在内存里临时做成一个哈希表（类似 Map）。

当它遍历新节点时，拿着新节点的 key 去哈希表里“啪”的一下查（O(1) 速度），一瞬间就能知道这个节点在旧列表里到底在哪个位置、有没有被复用

如果你不写 key，Vue 就没办法做哈希表，只能苦哈哈地用嵌套循环去死扣，性能当场暴跌。

vue框架本质是虚拟dom，diff算法来实现找不同之后再做出正确的修改，v-if不写：key来对应hash表的index的话只能嵌套，没有性能 10分钟之前写什么玩意啊，现在就看不懂了真难啊

diff算法有点像查找算法 key纯储在哈希表里面 查找的时候是哈希查找 很快 就是这个意思 所以要写在v-for 里面写:key
