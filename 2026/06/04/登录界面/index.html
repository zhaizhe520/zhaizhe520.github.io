登录界面

为什么只要一张表？

“注册”和“登录”其实是对同一批数据的两种不同操作：

注册（Create）： 往表里插入一条新的用户数据（比如写入用户名和密码）。

登录（Read）： 从表里查询有没有匹配的用户名和密码。

如果分成两个表，当一个新用户注册时，你得同时往“注册表”和“登录表”各写一条数据；以后用户修改密码，你还得同时更新两张表，不仅浪费空间，还极易导致数据不一致（比如 A 表更新了，B 表没更新成功）。

基础用户表（users）该怎么设计？
刚开始学全栈，你的核心用户表只需要包含以下几个基础字段（以 MySQL 为例）：

字段名 (Column) 类型 (Type) 作用 (Description) 备注 (Notes)
id INT 唯一标识 主键，自增 (Primary Key, Auto Increment)
username VARCHAR 用户名/账号 唯一约束 (Unique)，不允许重复
password VARCHAR 密码 绝对不能存明文！必须加密（如 bcrypt）
created_at TIMESTAMP 注册时间 自动生成，方便后续统计

全栈实现时的业务流程

两者的前端页面是分开的，但后端的核心逻辑都是围绕这张 users 表展开：

📝 注册流程 (Sign Up)
前端输入用户名和密码，发送给后端。

后端先去 users 表里查一下：SELECT * FROM users WHERE username = ‘输入的用户名’。

如果查到了： 返回错误“用户名已存在”。

如果没查到： 把密码用加密算法（如 bcrypt）打散，然后插入新数据：INSERT INTO users (username, password) VALUES (…)。

🔑 登录流程 (Sign In)
前端输入用户名和密码，发送给后端。

后端去表里查这个用户：SELECT * FROM users WHERE username = ‘输入的用户名’。

如果没查到： 返回错误“用户不存在或密码错误”。

如果查到了： 取出表里的加密密码，和用户输入的密码进行比对。比对成功则登录成功，生成 Token（如 JWT）返回给前端。

💡 全栈新手小贴士：
在写密码字段时，千万不要直接把用户输入的 123456 存进数据库。作为全栈，第一天就要养成安全意识，去搜一下你所用后端语言的 bcrypt 库，用它来加密和校验密码。

企业放 JWT 令牌

（经典双 Token 机制）

令牌名称存活时间 (Expires)作用 (Role)存放位置 (Storage)访问令牌 (Access Token)短（如 30 分钟 ~ 2 小时）真正用来调接口、拿数据的通行证。

因为时间短，哪怕被黑客偷走，半小时后也自动失效了。LocalStorage / Memory刷新令牌 (Refresh Token)长（如 7 天 ~ 30 天）不用来调数据。

它的唯一作用是：当“访问令牌”过期时，悄悄找后端“续命”换一个新的访问令牌。仅限 HttpOnly Cookie (极安全)
